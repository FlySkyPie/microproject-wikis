tl;dr  [Package by Feature](<#Package by Feature>)

項目結構有很多種，但最好的結構是一致、直觀且沒有意外的。

許多示例項目和教學按檔案類型（如crud、routers、models）劃分項目，這種方式對於微服務或範圍較小的項目很有效。但是，這種方法並不適合我們這個包含許多領域和模組的單體應用。

我發現對於這類情況，更具可擴展性和可演進性的結構是受Netflix的[Dispatch](https://github.com/Netflix/dispatch)啟發，並做了一些小修改。

│   │   └── utils.py
│   ├── aws
│   │   ├── client.py  # 用於外部服務通訊的客戶端模型
│   │   ├── schemas.py
│   │   ├── config.py
│   │   ├── constants.py
│   │   ├── exceptions.py

│   │   ├── schemas.py

1. 將所有領域目錄儲存在`src`資料夾中
   1. `src/` - 應用的最高等級，包含通用模型、組態和常數等。
   2. `src/main.py` - 項目的根檔案，用於初始化FastAPI應用

2. 每個包都有自己的路由、模式、模型等。
   1. `router.py` - 每個模組的核心，包含所有端點
   2. `schemas.py` - 用於pydantic模型
   3. `models.py` - 用於資料庫模型
   4. `service.py` - 模組特定的業務邏輯
   5. `dependencies.py` - 路由依賴項
   6. `constants.py` - 模組特定的常數和錯誤程式碼
   7. `config.py` - 例如環境變數
   8. `utils.py` - 非業務邏輯函數，例如響應規範化、資料豐富等
   9. `exceptions.py` - 模組特定的異常，例如`PostNotFound`、`InvalidUserData`

3. 當包需要其他包的服務、依賴項或常數時，使用顯式的模組名匯入
```python
from src.auth import constants as auth_constants
from src.notifications import service as notification_service
from src.posts.constants import ErrorCode as PostsErrorCode  # 以防每個包的constants模組中都有標準的ErrorCode
```
